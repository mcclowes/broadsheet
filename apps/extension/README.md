# Broadsheet — browser extension

Minimal save-current-tab extension. Manifest v3. Talks to the Broadsheet web
app via `POST /api/articles` using the user's existing cookie session.

Builds for three browsers from one shared source tree:

- **Chrome / Edge / Brave / Opera** — published at
  <https://chromewebstore.google.com/detail/broadsheet/joflmcpipjmffhgonneneafllkfppdld>
- **Firefox** — built from `manifest.firefox.json` overrides (event-page
  background, gecko id). Upload to [addons.mozilla.org](https://addons.mozilla.org/).
- **Safari** — wrapped via `xcrun safari-web-extension-converter` into a
  macOS/iOS app bundle, submitted through App Store Connect.

## Auth model

Cookie-based. The extension calls `POST {baseUrl}/api/articles` with
`credentials: "include"`. Because the manifest grants host permission for
the production URL (and optionally for localhost / other hosts via the
options page), the browser attaches cookies from the target origin and
bypasses CORS.

**You must be signed in to Broadsheet in the same browser profile.** If not,
the API returns 401 and the extension surfaces "Not signed in — open
Broadsheet and sign in first."

## Files

- `manifest.json` — base MV3 manifest (Chromium). Action, service-worker
  background, host permissions, keyboard command, icons.
- `manifest.firefox.json` — Firefox-specific overrides. Deep-merged onto the
  base: swaps `background.service_worker` for an event-page `background.scripts`
  and sets `browser_specific_settings.gecko.id`.
- `manifest.safari.json` — Safari-specific overrides (minimum Safari version).
  The base service-worker background is inherited unchanged; Safari 16.4+
  supports MV3 service workers.
- `background.js` — reads active tab URL, injects a content script to grab the
  rendered DOM, POSTs `{ url, html }` to `/api/articles`, surfaces a browser
  notification. Uses the `chrome.*` namespace, which is polyfilled by both
  Firefox and Safari, so the same file runs unchanged in all three browsers.
- `popup.html` / `popup.js` — one-click save button + status.
- `options.html` / `options.js` — set Broadsheet base URL (default: production;
  change for dev or self-hosted instances).
- `icons/` — 16/48/128 PNG icons, procedurally generated by
  `scripts/generate-icons.mjs`.
- `scripts/generate-icons.mjs` — regenerates the PNG icons from a pixel
  function. No deps.
- `scripts/package.mjs` — packages the extension for Chrome, Firefox, and
  Safari. See "Building a release".

The privacy policy lives in the web app at
[`/privacy`](../../src/app/privacy/page.tsx) and is served publicly at
<https://broadsheet.marginalutility.dev/privacy>. Link to that URL from every
store listing.

## Rendered DOM capture

The extension injects a one-liner via `chrome.scripting.executeScript` to
read `document.documentElement.outerHTML` from the active tab and sends it
as the `html` field. The server parses this directly with Readability,
bypassing `fetchAndParse`. This means:

- Paywalled and auth-gated pages save whatever is rendered in the user's browser.
- Client-side-rendered pages (SPAs) are captured after hydration.
- No SSRF surface for this path — the server never fetches the URL when `html` is present.
- Restricted pages (chrome://, the Web Store, PDFs) can't be scripted; we fall back to URL-only and let the server try to fetch.
- Pages with rendered HTML over 4 MB fall back to URL-only.

## Development

### Chrome / Edge / Brave / Opera

1. Run the web app: `npm run dev` (in repo root).
2. Open Chrome → `chrome://extensions`.
3. Toggle **Developer mode** (top right).
4. **Load unpacked** → select `apps/extension/`.
5. Click **Settings** in the popup and change the base URL to
   `http://localhost:3000`. Chrome will prompt to grant the localhost host
   permission the first time.
6. Pin the extension. Click the toolbar icon to save, or press **⌘⇧S** /
   **Ctrl+Shift+S**.

### Firefox

The base `manifest.json` uses a service-worker background block that older
Firefox versions reject, so you need the Firefox build before loading
unpacked.

1. `npm run extension:package -- --target=firefox` from the repo root. This
   stages the merged manifest + shipped files and produces
   `apps/extension/dist/broadsheet-extension-firefox-<version>.zip`.
2. Open Firefox → `about:debugging#/runtime/this-firefox`.
3. **Load Temporary Add-on…** → pick the zip (or the staged directory; if
   you prefer hot-reload, unzip first and point at its `manifest.json`).
4. Same settings flow as Chrome to point at `http://localhost:3000`. Firefox
   will prompt for the optional host permission on first save.

Temporary add-ons are unloaded on browser restart — this is expected for dev.

### Safari

Safari only loads WebExtensions that have been wrapped in a native app via
Xcode. You need macOS + Xcode + Apple Developer account to run the
development build locally.

1. `npm run extension:package -- --target=safari` from the repo root. This
   stages the merged manifest + shipped files under
   `apps/extension/dist/broadsheet-extension-safari-<version>/`.
2. Feed that directory to the converter:
   ```bash
   xcrun safari-web-extension-converter \
     apps/extension/dist/broadsheet-extension-safari-<version> \
     --project-location apps/extension/dist/safari-xcode \
     --bundle-identifier dev.marginalutility.broadsheet \
     --no-open
   ```
3. Open the generated `.xcodeproj` in Xcode, select the **Broadsheet**
   scheme, and **Run**. Safari launches with the extension installed.
4. Safari → **Settings → Extensions** → enable Broadsheet. Grant it
   permission on the sites you want to save from.
5. Same settings flow as Chrome to point at `http://localhost:3000`.

Re-run the converter whenever you change `background.js`, `popup.*`,
`options.*`, or the manifest. Xcode will rebuild incrementally.

## Building a release

```bash
# all three targets (default)
npm run extension:package

# or a single target
npm run extension:package -- --target=chrome
npm run extension:package -- --target=firefox
npm run extension:package -- --target=safari
```

This produces, under `apps/extension/dist/`:

- `broadsheet-extension-chrome-<version>.zip` — Chrome Web Store upload.
- `broadsheet-extension-firefox-<version>.zip` — AMO upload.
- `broadsheet-extension-safari-<version>/` — unzipped source tree, feed to
  `xcrun safari-web-extension-converter` on macOS (see above).

To bump the version:

1. Edit `apps/extension/manifest.json` → `"version"`.
2. Commit. Tag with `extension-vX.Y.Z` and push the tag — the
   `extension-release` workflow (`.github/workflows/extension-release.yml`)
   packages Chrome and Firefox and attaches both zips to a GitHub release
   automatically. Safari is built manually on macOS (see above).

Icons are checked in. If you change the icon design, run
`npm run extension:icons` to regenerate them.

## Publishing

### Chrome Web Store

First-time checklist:

1. **Developer account.** Register at
   <https://chrome.google.com/webstore/devconsole/> (one-time $5 USD fee).
2. **Confirm the privacy policy is live** at
   <https://broadsheet.marginalutility.dev/privacy>. The source lives in
   `src/app/privacy/page.tsx`; deploy the web app first so the Web Store
   reviewer can reach it.
3. **Create a new item** in the dev console and upload the Chrome zip from
   `npm run extension:package -- --target=chrome`.
4. **Fill in the listing:**
   - Short description (132 chars max): "Save the current tab to your Broadsheet read-it-later library."
   - Full description: lift from the root `README.md`.
   - Category: **Productivity**.
   - Screenshots: 1280×800 or 640×400 PNG/JPEG, at least one.
   - Icon: the 128×128 from `icons/icon128.png` is used automatically.
5. **Justify permissions** in the privacy practices tab:
   - `activeTab` + `scripting` — read the active tab's URL and rendered HTML on explicit user action (click / keyboard shortcut) to save it to the user's library.
   - `storage` — persist the `baseUrl` preference.
   - `notifications` — show save-result toasts.
   - Host permission for `broadsheet.marginalutility.dev` — deliver the save request with the user's existing cookie session.
   - Remote code: **none**.
6. **Link the privacy policy URL** from step 2.
7. **Submit for review.** Review usually lands within a few days; rejections typically cite the permission justifications, so keep them tight.

### Firefox Add-ons (AMO)

First-time checklist:

1. **Developer account.** Register at
   <https://addons.mozilla.org/developers/> (free).
2. Confirm the privacy policy is live (same URL as Chrome).
3. **Submit a new add-on** and upload the Firefox zip from
   `npm run extension:package -- --target=firefox`.
4. Choose **"On this site"** distribution. AMO signs and hosts the xpi.
5. Provide the source listing metadata:
   - Categories: Productivity, Bookmarks.
   - Summary: same as Chrome short description.
   - Description: lift from the root `README.md`.
   - Icon: uses `icons/icon128.png`.
6. Permission justifications are the same as Chrome.
7. **Submit for review.** AMO's automated validator runs first; a human
   reviewer typically follows within days.

Subsequent releases: bump `manifest.json` version, push an `extension-vX.Y.Z`
tag, then upload the matching zips from the resulting GitHub release.

### Safari (App Store)

Safari extensions ship as macOS (and optionally iOS) apps built with Xcode.
This requires an active **Apple Developer Program** membership ($99/year).

First-time checklist:

1. Build the wrapped project with `xcrun safari-web-extension-converter` (see
   "Development → Safari" above).
2. In Xcode: set the bundle identifier (e.g.
   `dev.marginalutility.broadsheet`), set your development team, set a
   version + build number matching `manifest.json`.
3. **Archive** via Xcode (Product → Archive), then distribute via App Store
   Connect.
4. In App Store Connect: create a new app record, fill in the listing (same
   description / screenshots as Chrome), and attach the archive.
5. Privacy policy URL: same as Chrome / Firefox.
6. **Submit for review.** Apple's review is typically the slowest of the
   three — budget a week or two.

Subsequent releases: re-run the converter against the new Safari bundle,
bump the Xcode version + build, archive, and re-submit.

## Known limits

- Icons are placeholder-quality (solid accent colour + horizontal lines).
  Replace with designed assets before serious launch.
- Safari builds require manual Xcode work (can't run in Linux CI).
- The Firefox gecko id (`broadsheet@marginalutility.dev` in
  `manifest.firefox.json`) must remain stable across releases — changing it
  means users re-install as a new add-on.
